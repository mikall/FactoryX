## Risk Classification

| Severity | Description | Response Time |
|----------|-------------|---------------|
| **CRITICAL** | Blocks sprint or threatens project viability | Immediate — escalate to PM + Architect |
| **HIGH** | Significant impact, requires active mitigation | Current sprint |
| **MEDIUM** | Moderate impact, monitor and plan mitigation | Next 1-2 sprints |
| **LOW** | Minor impact, accept or address opportunistically | Backlog |

## Probability Scale

| Level | Description |
|-------|-------------|
| **HIGH** | Very likely to occur (>70%) |
| **MEDIUM** | May occur (30-70%) |
| **LOW** | Unlikely (<30%) |

### Risk Categories

- **Technical**: architecture, performance, scalability, integration
- **Security**: vulnerabilities, data protection, compliance
- **Business**: scope creep, requirement changes, stakeholder alignment
- **Dependencies**: third-party services, external APIs, open-source components
- **Team**: knowledge gaps, capacity, tool expertise
- **Regulatory**: GDPR, industry-specific compliance, legal

## Risk Review Protocol

1. **Sprint Planning** (Phase 2 - Technical Table): each agent identifies domain-specific risks
2. **Sprint Closure**: review all active risks, update status, add new risks discovered during development
3. **Escalation**: CRITICAL risks trigger immediate PM + Architect review
